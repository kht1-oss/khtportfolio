# 맛킷리스트 글 수정 기능 — 설계 문서

작성일: 2026-06-22
대상: 기존 맛킷리스트 사이트에 "본인 글 수정" 기능 추가

## 한 줄 요약
작성자 본인이 자기 글의 모든 항목(음식이름·작성자·가격·가게·본문·사진)을 수정할 수 있게
한다. 글쓰기 창을 재사용하고, 추천수·소유권·작성시각은 유지한다.

## 확정된 결정
| 항목 | 결정 |
|---|---|
| 수정 항목 | 전체(음식이름·작성자·가격·가게·본문·사진) |
| 수정 권한 | 작성자 본인만 (관리자는 기존대로 삭제만) |
| 사진 | 새로 안 고르면 기존 유지, 고르면 교체 |
| 추천수/소유권/시각 | 수정해도 유지(초기화 안 됨) |
| UI | 글쓰기 창 재사용(수정 모드) |

## 1. 수정 버튼
- 카드 하단(footer)에 **✏️ 수정** 버튼 추가. **작성자 본인일 때만** 표시.
- 기존 🗑️ 삭제(본인+관리자)와 나란히. 수정은 본인만이므로 별도 권한 함수로 표시 제어.

## 2. 수정 모드(글쓰기 창 재사용)
- ✏️ 클릭 → 글쓰기 모달이 해당 글의 현재 값으로 채워져 열림(음식이름·작성자·가격·가게·본문).
- 모달 제목 "맛집 수정", 제출 버튼 "수정"으로 변경. 닫으면 다시 작성 모드로 복귀.
- 사진 입력은 선택: 안 고르면 기존 `imageUrl` 유지, 새로 고르면 압축 후 교체.
- 모달에 안내 문구: "사진을 바꾸려면 새로 선택(안 하면 기존 사진 유지)".

## 3. 저장 동작
- 수정 모드 제출 → `updateDoc`로 내용 필드만 갱신:
  `foodName, author, price(number), storeName, body, imageUrl`.
- **변경하지 않는 필드**: `votes`, `voters`, `ownerUid`, `createdAt`. (추천 유지)
- 입력 검증은 기존 `validatePostInput` 사용하되, 수정 모드에서는 기존 사진이 있으므로 `hasImage`를 true로 처리.

## 4. 보안 규칙
- 기존 `posts` `update` 규칙은 "추천(votes+1, voters+uid)"만 허용. 여기에 **소유자 내용 수정 경로**를 OR로 추가:
  - `request.auth.uid == resource.data.ownerUid`
  - 바뀐 키가 `['foodName','author','price','storeName','body','imageUrl']` 안에만 있음(`affectedKeys().hasOnly(...)`)
  - `votes`/`voters`는 그대로(unchanged), 타입 검증(문자열/숫자)
- 즉 추천 경로 또는 소유자-내용수정 경로 중 하나를 만족할 때만 update 허용.

## 5. 순수 함수(테스트 대상)
- `canEditPost(post, user) -> boolean`: `user`가 있고 `post.ownerUid === user.uid`면 true.
  (관리자 예외 없음 — 본인만.)

## 6. 변경 대상 파일
- `mukgit.utils.js` — `canEditPost` 추가(+테스트)
- `mukgit.html` — 글쓰기 모달에 사진 안내 문구(수정 시), 필요 시 제목/버튼 식별 유지
- `mukgit.js` — 수정 버튼 렌더(본인만), 수정 모드 열기/채우기, 제출 시 updateDoc, 모달 제목/버튼 토글
- `firestore.rules` — 소유자 내용 수정 경로 추가

## 7. 사용자 수동 작업
- Firestore 규칙 재게시(수정 경로 추가 때문). 구현 후 안내.

## 8. 테스트 (수동 확인)
- 본인 글에 ✏️ 보이고, 남의 글엔 안 보임(관리자도 남의 글엔 ✏️ 없음)
- ✏️ → 기존 값 채워진 창, 일부 수정 후 "수정" → 목록·상세창에 반영
- 사진 안 고르면 기존 사진 유지, 새로 고르면 교체
- 수정 후에도 추천수·추천 상태 유지
- 규칙: 비소유자가 내용 수정 시도 차단(콘솔/직접 시도)

## 9. 범위 밖(YAGNI)
- 수정 이력/되돌리기, 관리자의 타인 글 수정, 부분 필드별 인라인 편집
